# Map Merge into Miro Plan

## 1) 목적
- 현재 `MiroSystem` 하위에 생성되는 미로 결과를 `Map/Plane` 기준으로 적용 가능한 구조로 전환한다.
- `ChangeMap`의 맵 전환 흐름(머티리얼 + 스폰 위치)과 미로 저장/로드 흐름을 연결한다.
- 코드 구현 전에, 충돌 없는 통합 절차와 저장 정책을 확정한다.

## 2) 현재 구조 요약
- `MiroLineRenderer`는 큐브 세그먼트를 생성하고 기본 부모는 `MiroGeneratedLines`(`MiroSystem` 하위)다.
- `ChangeMap`은 `Plane Renderer.sharedMaterial`을 `mapMaterials[]`로 교체하는 방식이다.
- `MiroMazePersistence`는 `MiroMazeData(JSON)`만 저장하며, 머티리얼/텍스처/맵 목록은 저장하지 않는다.
- `VirtualLineSensor`는 히트한 오브젝트의 머티리얼 색상/텍스처(uv 샘플)로 검정 라인을 판정한다.

## 3) 핵심 결정 (권장)
- 통합 기준은 "Plane 중심"으로 잡는다.
- 미로를 Plane 위 오브젝트로만 올리는 방식보다, 최종적으로는 "Plane 텍스처/머티리얼"로 만드는 방식을 우선 채택한다.
- 이유:
1. `ChangeMap`이 이미 머티리얼 전환 모델이라 자연스럽게 결합된다.
2. 센서는 Plane 한 장만 Raycast하면 되므로 충돌/레이어 관리가 단순해진다.
3. 저장 시 `MiroMazeData + 텍스처 파일 경로` 조합으로 재구성이 쉽다.

## 4) 적용 방식 비교

### A안: 기존 라인 오브젝트를 Plane 자식으로만 이동
- 장점: 현재 구조 변경이 가장 작다.
- 단점: `ChangeMap.mapMaterials[]`와 직접 연결이 약하고, 맵 슬롯 관리가 이원화된다.
- 단점: 오브젝트 수가 많아질수록 관리/성능 부담이 커진다.

### B안(권장): 미로를 Texture2D로 래스터화해서 Plane 머티리얼에 반영
- 장점: `ChangeMap` 구조와 1:1 대응된다.
- 장점: 저장/로드 시 머티리얼을 재생성해 같은 맵을 복원하기 쉽다.
- 장점: 센서가 `hit.textureCoord` 기반으로 안정적으로 라인을 읽을 수 있다.
- 단점: 텍스처 생성/좌표 매핑 규칙을 먼저 확정해야 한다.

## 5) 통합 설계안 (권장안 기준)

### 5.1 렌더링 책임 분리
- `MiroAlgorithm`: 미로 데이터 생성만 담당.
- `MiroMapBaker`(신규 개념): `MiroMazeData` -> `Texture2D`(흰 배경/검은 주행 라인) 변환.
- `Map/ChangeMap`: 생성된 텍스처를 머티리얼에 넣어 Plane에 적용.

### 5.2 맵 카탈로그 개념 도입
- `ChangeMap`의 `mapMaterials[]`는 "정적 기본 맵"으로 유지.
- 런타임 생성 맵은 별도 "동적 맵 카탈로그"(리스트)로 관리.
- 최종 맵 선택 인덱스는 `정적 + 동적` 통합 인덱스로 매핑.

### 5.3 스폰 위치 정책
- 미로별 시작/회전값을 `SpawnPose`로 함께 저장.
- 동적 맵 추가 시 스폰값도 같은 슬롯으로 같이 등록.
- `ButtonRestart`는 기존 `ChangeMap.TryGetCurrentSpawnPose` 흐름을 그대로 사용.

## 6) 질문 2에 대한 답변 (런타임 가능 여부)

### 6.1 "저장하면 ChangeMap 배열에도 추가" 런타임 가능?
- 가능하다.
- 단, "배열 필드 수정"은 런타임 메모리에서만 유지된다.
- 빌드 실행 종료 후에도 유지하려면 별도 저장 파일(JSON/DB)이 반드시 필요하다.

### 6.2 "머티리얼을 미리 생성하고 저장하면 될까?"
- 고정된 맵 개수라면 가장 안정적인 방법이다.
- 사용자 생성 미로가 계속 늘어나는 구조라면 "사전 생성만"으로는 한계가 있다.
- 권장:
1. 기본 맵은 에셋 머티리얼(사전 생성) 사용.
2. 생성형 미로는 런타임 머티리얼 + 텍스처 파일(PNG) 저장 후 재로딩.

## 7) 저장/로드 전략

### 7.1 Local 기준 (1차)
- 저장 대상:
1. `MiroMazeData`
2. 베이크된 텍스처 파일 경로
3. 맵 슬롯 메타데이터(맵 이름, 생성시각, 스폰 위치)
- 시작 시 메타데이터를 읽고 동적 머티리얼을 재구성해 카탈로그 복원.

### 7.2 DB 기준 (2차)
- DB에는 머티리얼 자체가 아니라 "미로 데이터 + 텍스처(또는 생성 파라미터)"를 저장.
- 클라이언트는 로드 후 머티리얼 인스턴스를 재생성.
- 현재 `enableRemoteDb` 토글 구조를 그대로 활용해 Local fallback 유지.

## 8) 단계별 작업 계획 (코드 작업 전)
1. Plane 통합 방식 최종 확정: A안/B안 중 B안 채택 여부 승인.
2. 데이터 스키마 확정: 동적 맵 메타(JSON) 필드 정의.
3. 좌표/해상도 규칙 확정: `mazeSize -> textureSize`, 라인 두께(px), UV 기준.
4. ChangeMap 확장 설계: 정적 배열 + 동적 카탈로그 병합 인덱스 정책.
5. 저장 흐름 설계: Generate -> Bake -> Slot 등록 -> Save 메타/텍스처.
6. 로드 흐름 설계: 앱 시작 시 슬롯 복원 -> 머티리얼 재생성 -> Plane 적용.
7. 센서 검증 계획: 검정 임계값/텍스처 필터/타일링 값 검증.
8. 실패 복구 계획: 텍스처 파일 유실 시 슬롯 비활성화 + 안내 로그.

## 9) 검증 체크리스트
- [ ] Generate 후 Plane에 즉시 적용된다.
- [ ] 맵 전환 버튼으로 정적/동적 맵 모두 순환된다.
- [ ] 맵 전환 시 차량 스폰 위치가 올바르게 변경된다.
- [ ] 저장 후 재실행해도 동적 맵이 복원된다.
- [ ] 센서가 Plane 텍스처의 검정 라인을 안정적으로 감지한다.
- [ ] Local/Remote 저장 모드 전환 시 동작이 깨지지 않는다.

## 10) 리스크와 대응
- 리스크: 런타임 생성 머티리얼만 사용 시 재실행 후 유실.
- 대응: 머티리얼이 아닌 "원본 데이터 + 텍스처 파일 + 메타" 저장.

- 리스크: 텍스처 해상도 부족으로 센서 판독 흔들림.
- 대응: 최소 해상도 기준과 라인 두께(px) 하한선을 사전 정의.

- 리스크: `ChangeMap` 배열 중심 구조와 동적 추가 구조 충돌.
- 대응: 배열은 기본 맵 전용, 동적은 별도 리스트로 병합 조회.

## 11) 최종 제안
- 단기: `MiroSystem` 생성 결과를 Plane 기반(텍스처/머티리얼)으로 전환.
- 중기: `ChangeMap`에 동적 맵 카탈로그를 붙여 저장/불러오기 일관성 확보.
- 장기: Remote DB에는 미로 데이터 중심으로 저장하고 클라이언트에서 머티리얼 재구성.

## 12) 방향 전환 결정: 런타임 맵 스케일을 `2,1,2`로 통일

### 12.1 결정
- 런타임 생성 맵의 기준 스케일을 기본 맵과 동일하게 `2,1,2`로 맞춘다.
- 목표는 "기본맵과 런타임맵 전환 시 선 굵기 체감 차이 최소화"다.

### 12.2 적용 범위
- Generate 미리보기에서 보이는 런타임 Ground/Plane.
- Save 후 등록되는 런타임 맵 슬롯.
- Load/복원으로 재생성되는 런타임 맵.

### 12.3 기대 효과
- 맵 전환 시 상대 굵기/밀도 차이 감소.
- 차량 센서 판독 편차 감소.
- 디버깅 시 시각 기준 일관성 향상.

## 13) 원인 요약과 제약

### 13.1 원인 요약
- 현재 기본 Plane은 `2,1,2`, 런타임 생성 Plane은 더 큰 스케일로 생성된다.
- 같은 라인/텍스처라도 기준 면적이 다르면 상대 굵기가 달라 보인다.

### 13.2 제약 사항
- 단순히 스케일만 맞춰도, 머티리얼 타일링/오프셋 차이가 있으면 체감 차이는 남을 수 있다.
- 향후 기본 Plane 스케일이 바뀌면 고정값 하드코딩은 유지보수 리스크가 있다.

## 14) 구현 원칙 (이번 작업 기준)

### 14.1 1차 원칙
- 런타임 맵은 우선 `2,1,2`로 강제 통일한다.
- 맵 전환 기준은 동일 Plane 좌표계(기본 MapPlane 기준)를 사용한다.

### 14.2 2차 원칙
- 가능하면 고정 상수 대신 "기본 MapPlane Transform 복사" 방식으로 확장 가능성 확보.
- 스케일 통일과 함께 머티리얼 타일링/오프셋도 동일값 유지 정책을 적용한다.

2. Save 등록 시 생성되는 런타임 슬롯도 동일 스케일을 따른다.

### 15.3 머티리얼 매핑 일관화
1. 런타임 머티리얼의 타일링/오프셋을 기본맵 기준과 동일하게 맞춘다.

1. 순환 순서(`0->1->2->runtime...->0`)에서 기본/런타임 시각 차이를 비교한다.
2. 런타임 맵이 여러 개일 때도 동일한 체감 굵기를 유지하는지 확인한다.

### 15.5 센서 검증
1. 동일 속도/동일 코스로 기본맵과 런타임맵 판독 로그를 비교한다.
2. 코너/분기 구간에서 오검출 증가 여부를 점검한다.

## 19) 실제 코드 작업 순서 (구현용)

### 19.1 1단계: 기준 Plane 참조 고정
1. 수정 파일: `Assets/Scripts/Map/ChangeMap.cs`
2. 작업: 기준이 되는 MapPlane Transform/Renderer 참조가 항상 유효한지 확인(수동 연결 + 자동 탐색).
3. 목적: 런타임 쪽이 참조할 "단일 기준 Plane"을 명확히 만든다.
4. 완료 조건: Play 시작 시 `planeRenderer`가 null이 아니고, 실제 기준 Plane을 가리킨다.

### 19.2 2단계: 런타임 Ground 스케일 강제(`2,1,2`)
1. 수정 파일: `Assets/Scripts/Miro/MiroLineRenderer.cs`
2. 작업:
   - `CreateGroundPlane()`에서 런타임 Ground의 `localScale`을 `new Vector3(2f, 1f, 2f)` 기준으로 적용.
   - Plane primitive의 10x10 보정이 필요한 경우 `2/10`, `2/10` 형태로 변환해 적용.
3. 목적: Generate 미리보기 런타임 Ground가 기본맵과 동일 스케일 기준을 사용하게 한다.
4. 완료 조건: Generate 후 생성되는 `MiroGround`의 Transform Scale이 기준값으로 고정된다.

### 19.3 3단계: 런타임 Ground 위치/회전 정렬
1. 수정 파일: `Assets/Scripts/Miro/MiroLineRenderer.cs`
2. 작업:
   - 런타임 Ground의 Position/Rotation을 기준 Plane과 동일 축 기준으로 정렬.
   - 필요하면 `groundThickness`, `lineVerticalOffset`만 유지하고 기준 Plane 위/아래 오프셋만 적용.
3. 목적: 스케일뿐 아니라 좌표계 불일치로 생기는 체감 차이 제거.
4. 완료 조건: 기본맵 전환 직후/런타임 전환 직후 시각 기준점이 어긋나지 않는다.

### 19.4 4단계: Generate/Save/Load 경로 일관화
1. 수정 파일: `Assets/Scripts/Miro/MiroTestSceneController.cs`
2. 작업:
   - Generate 미리보기 등록, Save 등록, Load 복원 경로에서 동일한 기준(스케일/좌표계)을 쓰는지 점검.
   - 설정 플래그(`registerMapToChangeMapOnGenerate`, `persistGeneratePreviewToDisk`, `autoSaveOnGenerate`)가 현재 정책과 일치하는지 정리.
3. 목적: 특정 경로에서만 스케일이 달라지는 회귀 방지.
4. 완료 조건: Generate만 했을 때와 Save/Load 후의 시각 결과가 동일하다.

### 19.5 5단계: 씬 직렬화 값 동기화
1. 수정 파일: `Assets/Scenes/CreateBlock.unity`
2. 작업:
   - 정책 플래그 값을 실제 운영값으로 고정(예: Generate 미리보기 on, auto-save off 등).
   - 인스펙터에 남아 있는 과거 값이 코드 기본값을 덮어쓰지 않게 동기화.
3. 목적: "코드는 맞는데 씬 값 때문에 다르게 동작"하는 문제 제거.
4. 완료 조건: 씬 재오픈/재생성 후에도 동일 동작 유지.

### 19.6 6단계: 회귀 테스트 실행
1. 테스트 시나리오:
   - `Generate 1회` 후 기본맵(0~2)과 런타임맵(3+) 시각 두께 비교.
   - `Save -> Load -> 재시작` 후 동일 맵 재비교.
   - 맵 전환 순환(`0->1->2->runtime...->0`)에서 두께/센서 반응 확인.
2. 로그 확인:
   - `ChangeMap` 인덱스 순환 로그.
   - `VirtualCarPhysics`/`ButtonRestart` 로그가 클릭당 1회 흐름인지 확인.
3. 완료 조건: 두께 체감 차이 허용 범위 내 + 센서 오검출 증가 없음.

### 19.7 7단계: 후속 개선(선택)
1. 하드코딩 `2,1,2`를 유지할지, 기준 Plane Transform 복사 방식으로 일반화할지 결정.
2. `MapMergeintoMiroPlan.md`에 최종 선택과 근거를 기록.
3. 팀 공통 규약으로 스케일/UV/해상도 기준표를 고정한다.

## 20) 신규 이슈: 기본 맵만 선이 두껍게 보이는 문제

### 20.1 현상 정리
- 런타임 맵(베이크 텍스처)은 선이 얇고 균일하게 보인다.
- 기본 맵(`mapMaterials[0..2]`)은 같은 Plane에서도 선이 상대적으로 두껍게 보인다.

### 20.2 문제 지점(원인 후보)
1. **정적 맵 텍스처 품질/스타일 차이**
- 기본 맵 원본 이미지 자체의 선 두께가 이미 두껍거나, 해상도가 낮아 확대 시 두껍게 보일 수 있다.

2. **정적 맵 머티리얼 설정 차이**
- 기본 맵 머티리얼의 `mainTextureScale/Offset`, 셰이더, 색 보정, 감마 차이로 런타임맵과 시각 차이가 발생할 수 있다.

3. **텍스처 Import 설정 차이**
- Filter Mode, Compression, Max Size, MipMap 설정이 런타임 생성 텍스처와 다르면 두께/경계 선명도가 달라진다.

4. **파이프라인 불일치**
- 런타임맵은 `MiroMapBaker` 규칙(`pixelsPerCell`, 색상, 회전, wrap/filter)으로 생성된다.
- 기본맵은 같은 규칙을 거치지 않은 외부 에셋이므로 스타일이 일치하지 않는다.

### 20.3 핵심 결론
- 현재 문제는 "오브젝트 스케일"보다는 **정적 맵 텍스처/머티리얼 파이프라인이 런타임맵과 다르기 때문**일 가능성이 가장 높다.
- `ChangeMap`은 머티리얼만 교체하므로, 입력 에셋 차이가 그대로 보인다.

## 21) 수정 방향 (코드 작업 전 계획)

### 21.1 권장안 A (우선순위 높음)
- 기본 맵 3개도 런타임맵과 동일한 규칙으로 **재베이크/재생성**한다.
- 즉, 정적 맵도 `MiroMapBaker`와 같은 해상도 규약/색상 규약/회전 규약을 따르게 만든다.
- 장점: 가장 일관된 결과, 유지보수 용이.

### 21.2 대안 B (에셋 유지)
- 기존 기본 맵 이미지/머티리얼을 유지하되 아래 항목을 맞춘다.
1. 텍스처 Import: Max Size, Filter, Compression, MipMap
2. 머티리얼: 셰이더, Texture Scale/Offset, Color
3. 선 굵기 원본 이미지 자체 보정(그래픽 에셋 수정)
- 단점: 맵별 수동 튜닝 필요, 회귀 가능성 높음.

## 22) 상세 작업 순서 (문제 해결용)
1. **비교 기준 수집**
- 기본맵 3개와 런타임맵 1개를 같은 카메라 위치에서 캡처.
- 선폭(픽셀 기준) 샘플 구간 5개 측정해 평균값 기록.

2. **정적 에셋 설정 점검**
- `Assets/Resources/Map Image/*.mat`의 셰이더/텍스처 스케일/오프셋 확인.
- 각 텍스처 Import 설정(Filter/Compression/Max Size/MipMap) 기록.

3. **런타임 기준값 확정**
- `MiroMapBaker`의 `pixelsPerCell`, `textureFilterMode`, `textureWrapMode`, 회전 규칙을 기준값으로 고정.

4. **정규화 적용**
- 권장안 A: 기본맵 3개를 동일 베이크 규칙으로 재생성해 교체.
- 대안 B: 기본맵 에셋에 동일한 Import/Material 규칙을 수동 반영.

5. **센서 영향 확인**
- 시각 두께 수정 후 `VirtualLineSensor` 임계값이 과민/둔감해졌는지 재측정.

6. **회귀 테스트**
- `0->1->2->runtime->0` 순환에서 선 두께 편차 재측정.
- 저장/로드/재시작 후 동일 조건 재검증.

## 23) 완료 기준 (신규 이슈)
- 기본맵 3개와 런타임맵의 선폭 평균 편차가 허용 범위 내(사전 정의)다.
- 맵 전환 시 "기본맵만 두꺼워 보임" 현상이 재현되지 않는다.
- 센서 검출 정확도(이탈/오검출)가 수정 전 대비 악화되지 않는다.
